## 📌 产品定位

**AI 医嘱任务引擎**是一个面向康养中心的 AI 驱动医嘱闭环管理系统，覆盖医嘱从创建、AI 解析、任务拆解、护士执行到管理监控的全流程，服务**医生、护士、管理**三类角色，支撑 10 万用户规模。

系统核心价值：
- **医嘱解析自动化** — 医生自然语言输入，AI 自动提取结构化字段，解放填表负担
- **任务执行闭环** — 每条医嘱生成护士任务，扫码核验执行，结果自动回写
- **风险智能预警** — 执行异常实时上报，管理大屏 AI 预警，防患于未然

## 🔄 核心业务流程（7 步闭环）

```
① 输入        ② 解析            ③ 确认          ④ 拆解
医生自然语言  →  AI 自动提取     →  医生审阅      →  系统生成
/语音输入       结构化字段          置信度确认       护理/用药/检查任务
                NLP + 置信度评分    人机协同          任务模板 + 自动派发
     │
     ▼
⑦ 监控           ⑥ 回写           ⑤ 执行
管理层大屏    ←   执行结果自动    ←  护士扫码核验
+ AI 风险预警      同步至医嘱         腕带 + 药品双重核验
```

**三个关键闭环：**

| 闭环类型 | 链路 | 目的 |
|---------|------|------|
| **医嘱闭环** | 医生下单 → 护士执行 → 数据回写 | 每条医嘱完整追踪，无遗漏 |
| **质控闭环** | AI 解析 → 医生确认 → 错误上报 → 模型优化 | 持续提升 AI 解析准确率 |
| **监管闭环** | 执行数据 → 大屏监控 → AI 预警 → 管理干预 | 实时发现风险，及时介入 |

## 🏗️ 三端架构

### 👨‍⚕️ 医生端（Doctor Portal）— 4 个核心页面

| 页面 | 核心功能 | 设计亮点 |
|------|---------|---------|
| **医嘱创建** | 自由文本 + 语音输入医嘱 | 左侧输入 · 右侧 AI 实时预览 · 双栏布局 |
| **AI 解析确认** | 结构化字段审阅与修改 | 置信度可视化 · 分区展示（用药/护理/检查） |
| **医嘱详情** | 执行闭环全貌 | 执行率环形图 · Timeline 链路 · 风险提示 |
| **医嘱列表** | 医嘱状态管理 | 多维筛选（老人/时间/状态）· 快速搜索 |

**设计原则：**
- **降低认知负担** — AI 做结构化，医生只需确认，无需填写繁琐表单
- **透明可信** — 置信度明确展示，医生始终掌控最终决策权
- **闭环可视** — 医生可随时查看自己医嘱的执行状态，形成责任感

### 👩‍⚕️ 护士端（Nurse Task Center）— 4 个核心页面

| 页面 | 核心功能 | 设计亮点 |
|------|---------|---------|
| **今日任务列表** | 按病区查看当日全部任务 | 颜色标签区分（蓝=用药/绿=护理/紫=检查）· 红点高优先级 |
| **任务详情抽屉** | 查看老人信息与执行说明 | 侧滑抽屉 · 头像 + 简要病情 + 操作说明 |
| **扫码执行** | 扫腕带 + 药品完成核验 | 摄像头取景 · 自动弹出任务 · 双重校验防差错 |
| **异常上报** | 记录拒服药/药品缺失等异常 | 类型快选 + 备注 + 照片/语音上传 |

**扫码双重防差错机制：**

```
护士执行任务
      │
      ├── 扫描 1：老人腕带  →  确认身份（防止给错人）
      │
      └── 扫描 2：药品条码  →  确认药品匹配（防止给错药）
                │
         ✅ 两码均匹配  →  任务执行 → 自动回写医嘱系统
         ❌ 任一不匹配  →  系统阻断 + 立即提示，禁止继续执行
```

**设计原则：**
- **零歧义** — 每个任务指令明确，护士无需猜测下一步
- **防差错** — 扫码双重核验是强制流程，不是可选项
- **快速记录** — 语音输入护理记录，减少屏幕操作时间

## 💡 系统核心亮点

### 亮点一：AI 解析 + 人工确认协同模式

```
医生自然语言输入
        │
        ▼
AI 自动提取结构化字段
  ├── 药品名称      置信度 ████████░░ 82%  ← 高亮提示，需医生重点核查
  ├── 剂量          置信度 ██████████ 97%
  ├── 频次          置信度 ██████████ 95%
  ├── 疗程          置信度 █████████░ 91%
  ├── 护理项        置信度 ████████░░ 85%
  └── 复查项        置信度 ███████░░░ 76%  ← 高亮提示
        │
        ▼
医生审阅确认（可一键上报错误解析）
        │
        ▼
标注数据 → 模型再训练 → AI 越用越准（正向飞轮）
```

> 「AI 辅助 + 人工兜底」是医疗场景 AI 落地最合理的方式——医生既不需要填繁琐表单，也不会对黑盒 AI 完全失控。

### 亮点二：投诉链路重建（合规核心）

任何投诉发生后，可还原前后 **24 小时**完整任务链路：

```
[08:30] 用药提醒下发      → 护士确认接收
[09:15] 护士执行操作记录  → 双码扫描通过
[10:42] AI 风险预测触发   → 生成预警信号
[11:30] 投诉事件发生      ← 当前节点
        │
        └── 每个节点可点击查看详情，责任人清晰可查
```

> 链路重建的价值不只是「查责任」，更重要的是「改流程」——发现系统性问题（如某时段任务堆叠），提供排班优化的数据依据。

### 亮点三：全系统数据闭环

```
医生医嘱          →  任务调度系统    →  自动生成护士任务
护士执行记录      →  医嘱执行状态    →  更新医嘱完成度
执行异常上报      →  管理大屏        →  实时风险提示
AI 错误上报       →  模型训练数据    →  持续提升解析准确率
执行率统计        →  护理质量分析    →  支撑排班和培训决策
```

没有孤立的数据孤岛，每一条数据都在系统中形成完整回路。

## 🔐 三端权限边界

| 角色 | 可访问范围 | 限制 |
|------|-----------|------|
| **医生** | 自己开具的医嘱 | 不看护士任务详情，不访问管理大屏 |
| **护士** | 自己负责病区的任务 | 不能修改医嘱，只能上报异常 |
| **管理** | 全部数据（只读监控） | 无法直接修改医嘱或任务内容 |

> 清晰的权限边界避免越权操作带来的医疗风险，同时保持各角色专注度。

## 🎨 设计规范

| 要素 | 规范值 | 说明 |
|------|--------|------|
| 主色 | `#3A78F2` 医疗蓝 | 专业感、权威感，医疗行业通识色 |
| 辅色 | `#F5F7FA` 浅灰白 | 低认知负担，长时间使用不疲劳 |
| 圆角 | `8–12px` | 现代 SaaS 风格，亲和而不失专业 |
| 字号 | 大字号优先 | 适配老年护士/医生视力，减少误读 |

**标签颜色系统（全系统统一）：**

```
🔵 蓝色 = 用药任务
🟢 绿色 = 护理任务
🟣 紫色 = 检查任务
🔴 红色 = 紧急 / 危险操作
```

**医疗级可用性原则：**
- **低认知负担** — 每屏最多一个核心操作，避免高压环境误操作
- **可逆操作** — 重要操作（如结束医嘱）均有二次确认弹窗
- **状态透明** — 待执行/执行中/延误/完成，每种状态明确视觉区分
- **无障碍考量** — 大字号、高对比度、图标语义化，适配不同年龄段用户

## 🚀 快速开始

```bash
# 克隆项目
git clone [https://github.com/your-org/ai-medical-order-engine.git
cd ai-medical-order-engine](https://github.com/panglezym/AI-Medical-Order-Execution-Engine)

# 安装依赖
npm install

# 配置环境变量
cp .env.example .env
# 填写 NLP 服务地址、数据库连接、WebSocket 推送服务等配置

# 本地开发启动
npm run dev

# 构建生产版本
npm run build
```

**环境要求：**
- Node.js ≥ 18
- NLP 解析服务（可对接内部模型或第三方医疗 NLP API）
- WebSocket / SSE 推送服务（支持三端实时数据同步）
- 条码扫描 SDK（腕带 + 药品二维码 / 条形码）

## 🤝 贡献指南

欢迎提交 Issue 和 Pull Request。贡献前请注意：

- **安全关键路径（Safety-Critical）**：扫码核验、权限控制相关代码修改须经过完整的测试与审查，任何回归都可能导致医疗差错风险
